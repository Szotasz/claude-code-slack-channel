1. Holding the PR while a security blocker (the 6 transitive CVEs failing `bun audit`) was cleared via PR #163.
2. Rebasing the contributor's commit onto post-CVE main and resolving the CHANGELOG merge conflict (both PRs added entries under `[Unreleased]`).
3. Force-pushing the rebased commit back to the contributor's fork via the maintainer-edit channel (enabled by `maintainerCanModify=true` on the PR).
4. Leaving a courtesy comment explaining the rebase + force-push so the contributor could `git fetch && git reset --hard` if they had local work.

Net: external contributor goodwill preserved, attribution intact, no surprises.

## Lessons Learned

1. **Structural allowlists beat per-commit exemptions.** `.gitleaks.toml` regex for GHSA-XXXX-XXXX-XXXX now silently absorbs every future advisory citation — the previous fingerprint-by-fingerprint approach would have required a new PR for each security writeup.
2. **`bun update <dep>` adds direct dependencies, even when you wanted transitive resolution.** The right pattern for transitive CVE pinning is package.json `overrides`, not bumping individual transitives — the latter promotes them to direct deps in `dependencies`.
3. **CI red on a green PR is almost always upstream.** When PR #162 came in red, the cause was dep CVEs that hit main the same day from a different commit (30e620e). Always check whether main shares the failure before suspecting the contributor.
4. **A two-versions-behind README header is a calibration signal.** Line 1 of README still said `v0.8.0` going into this release — meaning v0.9.0 cut didn't update it either. Next release should treat the README header as a release-time mandatory edit, not a manual chore.
